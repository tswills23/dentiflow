# RECALL / REACTIVATION -- Complete Breakdown

## 1. Core Actions

### A. Accurate Recall Identification

**What must happen:**
- Identify overdue and unscheduled patients

**Why this is required:**
Recall only works when the list is correct. Messaging the wrong patients creates confusion and destroys trust.

**What breaks if missing:**
- Patients contacted after booking
- Wrong patients annoyed
- Staff disables recall systems

### B. Segmentation by Recency & Value

**What must happen:**
- Segment by time overdue and appointment type

**Why this is required:**
Patients overdue by different timeframes respond differently. Segmentation increases relevance.

**What breaks if missing:**
- Generic messages
- Lower response rates
- Recall blamed instead of messaging

### C. Multi-Touch Outreach

**What must happen:**
- SMS
- Email
- Optional call assist

**Why this is required:**
Recall requires persistence without annoyance.

**What breaks if missing:**
- One-and-done campaigns
- Inconsistent follow-through

### D. Frictionless Scheduling

**What must happen:**
- Simple booking path

**Why this is required:**
Recall patients already trust the practice. Friction reintroduces procrastination.

**What breaks if missing:**
- Abandoned scheduling
- Lost reactivation

### E. Clear Outcome Tracking

**What must happen:**
- Track contacted, booked, completed, no response

**Why this is required:**
Recall is a revenue recovery system, not a messaging task.

**What breaks if missing:**
- No ROI visibility
- No iteration

---

## 2. Ownership (Human vs System)

**System-owned:**
- Recall list generation
- Segmentation
- Outreach

**Human-owned:**
- Booking
- Exception handling

**Why this split is required:**
If recall depends on staff time, it will stop when schedules are full.

---

## 3. Failure Modes

- Dirty Lists: Wrong patients contacted.
- Single Message Campaigns: Recall attempted once then abandoned.
- No Urgency Framing: Messages give no reason to act.
- No Closed-Loop Tracking: No visibility into recovered revenue.

---

## 4. Recovery Logic

- No response → Continue outreach
- Partial response → Clarify and re-prompt
- Booked but no-show → Re-enter recall loop
- Decline → Pause and retry later

**Why this is required:**
Recall is non-linear. Systems must remember when humans forget.

---

## 5. Outputs

Recall must produce:
- Appointment scheduled
- Appointment completed
- Declined or deferred
- No response after X attempts

Anything else is noise.
